# Artifact Taxonomy

Этот документ фиксирует, куда складывать создаваемые артефакты проекта, чтобы структура оставалась предсказуемой по мере роста продукта.

## Базовое правило

Каждый новый артефакт должен попадать в каталог по своему назначению, а не в корень проекта.

Корень проекта используется только для:

- главных файлов проекта, если они появятся: `README.md`, лицензия, системные конфиги;
- верхнеуровневых каталогов;
- файлов, которые технически обязаны лежать в корне выбранного фреймворка или инструмента.

## Каталоги верхнего уровня

- `docs/` - проектная документация.
- `templates/` - шаблоны документов и рабочих артефактов.
- `src/` - исходный код приложения или сервисов.
- `tests/` - тесты.
- `scripts/` - вспомогательные скрипты для разработки, миграций, импорта, генерации и обслуживания.
- `config/` - конфигурационные файлы проекта, если они не обязаны лежать в корне.
- `assets/` - изображения, иконки, презентационные материалы, статические медиа.
- `data/` - локальные справочники, seed-данные, примеры данных, выгрузки для разработки.

## Документы

Документы внутри `docs/` классифицируются так:

- `docs/product/` - product brief, позиционирование, value proposition, MVP, scope, out of scope.
- `docs/requirements/` - требования, user stories, acceptance criteria, use cases.
- `docs/architecture/` - архитектура, компоненты, интеграции, ADR, data flow.
- `docs/planning/` - roadmap, task decomposition, milestones, release plan.
- `docs/research/` - рынок, пользователи, конкуренты, регуляторика, интервью, источники.
- `docs/operations/` - правила проекта, процессы, роли, качество, поддержка, delivery-подход.

## Код

Код хранится в `src/`.

Если проект станет монолитным приложением:

- `src/app/` - основная логика приложения;
- `src/api/` - API-слой;
- `src/domain/` - доменные модели и бизнес-правила;
- `src/infrastructure/` - интеграции, хранилища, внешние сервисы;
- `src/ui/` - frontend-интерфейс, если он живет в том же проекте.

Если проект станет multi-service:

- `src/services/<service-name>/` - отдельные backend-сервисы;
- `src/web/` - web-приложение;
- `src/shared/` - переиспользуемые типы, схемы и утилиты.

## Тесты

Тесты хранятся в `tests/`.

Рекомендуемая классификация:

- `tests/unit/` - модульные тесты;
- `tests/integration/` - интеграционные тесты;
- `tests/e2e/` - end-to-end сценарии;
- `tests/fixtures/` - тестовые фикстуры и примеры данных.

## Скрипты

Скрипты хранятся в `scripts/`.

Рекомендуемая классификация:

- `scripts/dev/` - локальная разработка и запуск;
- `scripts/data/` - импорт, экспорт, генерация данных;
- `scripts/migrations/` - миграции и разовые изменения данных;
- `scripts/ops/` - эксплуатационные и диагностические команды.

## Данные и ассеты

`data/` используется только для локальных и проектных данных, которые можно хранить в репозитории.

Не хранить в `data/`:

- реальные персональные данные детей, родителей, специалистов;
- ИИН, телефоны, медицинские заключения и чувствительные документы;
- production-выгрузки без обезличивания.

`assets/` используется для:

- изображений и иконок;
- презентационных материалов;
- схем и экспортов, если они не являются исходной документацией.

## Именование файлов

Используем:

- lowercase;
- kebab-case;
- понятные имена по содержанию.

Примеры:

- `product-brief.md`
- `user-stories.md`
- `system-architecture.md`
- `data-model.md`
- `mvp-roadmap.md`

## Правило для следующих артефактов

Перед созданием нового файла нужно выбрать категорию:

1. Продуктовый смысл - `docs/product/`.
2. Требования - `docs/requirements/`.
3. Архитектура или техническое решение - `docs/architecture/`.
4. План работ - `docs/planning/`.
5. Исследование или источники - `docs/research/`.
6. Процесс или правила проекта - `docs/operations/`.
7. Исполняемый код - `src/`.
8. Проверки - `tests/`.
9. Автоматизация - `scripts/`.
10. Конфигурация - `config/`.
11. Медиа и визуальные материалы - `assets/`.
12. Безопасные локальные данные - `data/`.

Если артефакт не попадает ни в одну категорию, сначала добавить правило классификации сюда, затем создавать файл.
